# 课程设计任务书

设计题目(范围)：结合实际问题，实现具有数据采集、异构数据集成、数据处理与分析、结果输出与展示等功能的综合实时大数据系统。

## 题目：智慧交通实时路况监测与拥堵分析系统设计与实现

### 1、课程设计目标

随着智慧城市建设的不断推进，城市交通系统每天都会产生海量实时数据，包括车辆 GPS 数据、道路车流量数据、交通传感器数据、导航平台路况数据以及交通事件数据等。这些数据具有典型的大数据“4V”特征，即数据规模大（Volume）、数据产生速度快（Velocity）、数据类型复杂（Variety）以及数据价值密度低（Value）。如何利用实时大数据处理技术对交通流进行实时采集、实时分析、实时预测与实时展示，已经成为智慧交通领域的重要研究方向。

实时交通数据处理不仅涉及海量数据的高速接入与处理，还涉及复杂事件检测、实时聚合计算、窗口分析、状态管理以及实时可视化等关键技术，是实时大数据处理技术的重要典型应用场景。通过构建智慧交通实时路况监测与拥堵分析系统，深入理解实时流处理系统的整体架构设计思想，掌握 Spark Streaming、Storm 或 Flink 等主流实时大数据处理框架的基本原理与工程实现方法，并能够将 Kafka、Flume、MySQL、Redis、ECharts 等相关技术综合应用于实际系统开发中。

本课程设计旨在通过一个完整的工程实践项目，使学生综合运用《实时大数据处理技术》课程中所学习的理论知识与实验技能，掌握实时数据采集、实时数据处理、流式计算、实时数据库存储、数据可视化展示以及系统优化等完整开发流程，培养学生分析问题、设计系统、工程实现和团队协作的能力，为后续的大数据分析、人工智能与智慧城市等方向的学习和研究奠定基础。

### 2、课程设计总体要求

本课程设计采用小组协作方式完成，每组人数为 6～7 人。各小组应根据题目要求，独立完成一个具有完整功能的实时大数据处理系统。系统应当能够完成交通数据的实时采集、实时处理、统计分析、结果存储以及可视化展示等功能，并能够体现实时大数据处理框架在海量流数据场景中的应用能力。

课程设计过程中，学生需要结合具体业务需求，对系统整体架构进行合理设计，并能够根据不同业务场景选择适当的大数据技术栈。系统必须至少采用 Spark Streaming、Storm 或 Flink 中的一种实时流处理框架实现核心实时计算功能，同时应当结合 Kafka、Flume 等消息中间件或数据采集工具构建完整的数据流转流程。

系统设计应**尽量体现实时性**、工程性和完整性，要求能够对交通流数据进行持续接收、持续处理与持续输出，并通过实时图表或地图动态展示系统分析结果。系统需具备基本的数据容错能力和一定的系统扩展能力，能够满足连续流式数据处理需求。

所有系统均需提供完整源代码、运行环境说明、数据库文件、课程设计报告以及系统演示。课程设计过程中严禁抄袭，一旦发现源代码、数据库结构、系统界面或报告内容存在大面积雷同，将按照课程设计不及格处理。

### 3、具体任务

 本课程设计围绕“智慧交通实时路况监测与拥堵分析系统”展开，要求结合实时大数据处理技术，实现一个能够持续处理交通流数据的综合实时分析平台。系统主要包括数据采集模块、数据清洗模块、实时流处理模块、数据存储模块、数据分析模块以及数据可视化展示模块。

#### （1）系统架构设计

课程设计开始阶段，各小组需要首先完成系统整体架构设计。系统架构设计应充分考虑实时数据处理系统的典型组成结构，包括数据采集层、消息队列层、实时计算层、数据存储层以及数据展示层等部分。需要根据系统功能需求分析数据流向、模块划分以及各组件之间的交互关系，并绘制完整的系统总体架构图。

系统架构设计过程中，需要明确说明各技术组件的具体作用。例如，Kafka 用于实时数据缓冲与消息传输，Spark Streaming 或 Flink 用于实时流数据计算，MySQL 或 Redis 用于统计结果存储，ECharts 或 Vue 用于数据可视化展示等。各小组需要结合所选框架的特点，分析系统在实时性、扩展性以及容错性方面的设计思路，并说明为什么选择当前技术方案。

系统架构图中应明确体现数据的输入、传输、处理、存储以及展示过程。要求系统整体逻辑清晰、模块划分合理，能够体现完整的实时大数据处理流程。

#### （2）实时交通数据采集

系统需要实现实时交通数据采集功能。交通数据可以来源于公开交通 API，也可以采用模拟数据生成方式完成。各小组可参考以下数据获取方式：

- 高德开放平台（交通态势 API）：主要可获取道路拥堵情况、实时交通状态、道路速度、行驶时间、路段拥堵等级等数据，典型接口包括交通态势查询：https://lbs.amap.com/api/webservice/guide/api-advanced/traffic-situation-inquiry
- 百度地图开放平台（实时路况 API）：主要可获取路段拥堵状态、平均速度、道路事件等数据，可通过以下接口访问数据：https://lbs.baidu.com/
- 国际开放交通数据平台OpenStreetMap（OSM）：主要可获取道路网络、路段信息、POI、路口数据等，适合做交通拓扑分析、路网可视化、热力图等。https://www.openstreetmap.org/?utm_source=chatgpt.com#map=3/35.59/134.38
- HERE Traffic API：主要可获取实时交通流、实时事件、拥堵等级、平均速度等数据。https://www.here.com/developer
- Chicago Traffic Tracker Dataset：主要可获取道路拥堵指数、平均速度、时间戳、区域信息等信息。https://data.cityofchicago.org/Transportation/Chicago-Traffic-Tracker-Historical-Congestion-Esti/77hq-huss/about_data
- Kaggle 交通流数据集：推荐数据集：PeMS Traffic Dataset，包括车流量、车速、占有率、时间序列等数据。https://pems.dot.ca.gov/?utm_source=chatgpt.com

由于真实交通数据接口通常存在访问频率限制和网络稳定性问题，因此课程设计推荐采用“历史交通数据 + Kafka 实时模拟”的方式构建实时数据流。

各小组可以从公开数据平台获取交通数据集，例如道路车流量数据、车辆速度数据、交通拥堵数据、交通事件数据、车辆GPS数据、路口通行数据、平均车速数据、交通事件数据、天气数据（可选）等。推荐使用 Kaggle、PeMS、高德开放平台等公开数据源。对于无法直接获取实时流数据的小组，可以利用 Python 编写数据模拟程序，按照固定时间间隔持续生成车辆位置、车速、道路编号、道路拥堵指数等信息，并通过 Kafka Producer 持续发送到 Kafka 消息队列中。

在数据采集模块中，理解流式数据与批量数据的区别，并掌握 Kafka Topic 创建、Producer 编程以及消息发送机制。系统应能够持续接收不断产生的数据流，并保证数据格式的一致性和完整性。

例如，可以设计如下格式的实时交通数据：

```(空)
{
  "road_id":"R102",
  "timestamp":"2026-05-20 08:15:20",
  "vehicle_count":120,
  "avg_speed":35,
  "congestion_index":0.72,
  "region":"海淀区"
}
```

系统应能够持续接收类似格式的数据并进入实时处理流程。

#### （3）数据清洗与预处理

由于实时交通数据在采集过程中可能存在缺失值、异常值、重复数据、格式不统一等、时空数据不一致等问题，因此系统需要实现数据清洗与预处理功能。

各小组需要利用 Spark、Flink 或 Storm 的相关 API 对原始数据进行处理。例如，对于车辆速度为负值的数据应视为异常数据进行过滤；对于时间字段格式不统一的数据应统一转换为标准时间格式；对于缺失字段的数据应进行补全或剔除；对于重复数据应进行去重处理。

数据预处理过程中，还需要对字段进行统一规范，例如将道路编号统一编码、将不同单位的速度值进行统一换算等。各小组需要展示数据清洗前后的对比效果，并说明所采用的数据预处理策略。

本部分重点考察对流数据处理流程的理解，以及对数据质量控制的重要认识。

#### （4）实时流数据处理与分析

实时流数据处理是本课程设计的核心内容。系统需要使用 Spark Streaming、Storm 或 Flink 实现交通数据的实时统计与分析功能。

系统==至少需要==实现以下基础功能：

①实时统计道路车流量；

②实时统计平均车速；

③实时计算道路拥堵指数；

④统计高峰时段流量变化；

⑤统计不同时间段交通变化趋势以及统计不同区域交通热度等。

系统应能够随着数据不断进入而持续更新统计结果。

各组可根据实际情况，选择完成以下==至少三项==进阶功能，鼓励大家利用状态管理机制实现持续状态计算，例如持续监控某道路是否连续超过指定拥堵阈值：

① 滑动窗口统计； 

② 实时 TopN 热点道路分析；

③ 异常车辆检测；

④ 交通事故预警；

⑤ 实时路径拥堵预测；

⑥ 实时区域拥堵排行；

⑦ 基于状态计算的持续拥堵检测。

为了体现实时流处理框架的特点，系统还应至少实现一种窗口计算机制。例如，可以使用滚动窗口统计最近 5 分钟内的平均车流量，也可以使用滑动窗口统计最近 10 分钟的道路平均速度变化趋势。窗口计算过程中需要正确理解窗口大小、滑动间隔以及时间语义等概念。

如果采用 Flink 实现系统，可进一步尝试使用状态管理、Watermark、Exactly-Once 等高级特性；如果采用 Spark Streaming，可尝试结合 Spark SQL 或 Structured Streaming 实现实时 SQL 分析。

本部分重点考察对实时流处理框架的掌握程度，以及对实时窗口计算和状态计算机制的理解。

#### （5）数据存储与查询

系统需要将实时计算结果保存到数据库中，以支持后续查询与展示功能。数据库可以采用 MySQL、Redis、HBase 或 Elasticsearch 等。

各小组需要根据业务需求设计合理的数据表结构。例如，可以设计实时道路状态表、交通统计结果表、异常事件记录表等。数据库设计过程中应考虑数据之间的逻辑关系，并合理设置主键、索引以及时间字段。

系统应能够支持实时数据写入，并能够根据时间范围、区域名称、道路编号等条件进行查询。需要说明数据库设计思路，并分析如何提高数据查询效率。

本部分重点考察对实时计算结果存储方式的理解，以及对数据库设计规范的掌握。

#### （6）实时数据可视化展示

系统需要开发交通数据可视化展示界面，用于实时展示道路运行状态和统计分析结果。

系统界面应能够动态展示实时车流量变化、道路平均速度变化、区域交通热力分布以及拥堵趋势等信息。推荐使用 ECharts、Vue、Flask、Streamlit 等工具实现可视化界面。

系统至少需要包含折线图、柱状图、饼图以及地图或热力图等可视化内容。例如，可以利用折线图展示道路平均速度变化趋势，利用柱状图展示不同区域车流量排行，利用地图热力图展示城市交通拥堵情况。

系统界面应尽量做到布局合理、界面清晰、交互友好，并支持用户根据时间、区域、道路等条件进行数据筛选与查询。鼓励学生实现实时自动刷新功能，使图表能够随着后台数据变化动态更新。

本部分重点考察数据分析展示能力以及系统前后端集成能力。

#### （7）系统测试与性能优化

系统开发完成后，需要对系统进行完整测试，包括功能测试、稳定性测试以及性能测试。

各小组需要测试系统在持续数据流输入情况下的运行效果，并观察系统是否存在数据延迟、数据丢失或程序崩溃等问题。对于系统运行过程中出现的性能瓶颈，应进行分析与优化。

优化内容可以包括 Kafka 参数优化、Spark/Flink 并行度优化、缓存机制优化、数据库索引优化以及代码结构优化等。学生需要对优化前后的系统运行效果进行对比分析，并说明所采用的优化策略。

本部分重点培养学生的工程实践能力与系统优化意识。

#### （8）技术路线选择

各小组可根据擅长的技术工具选择相应的技术路线，优先采用以下技术路线：Kafka + Spark Streaming + Spark SQL + MySQL + ECharts
