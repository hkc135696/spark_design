# 智慧交通实时路况监测与拥堵分析系统 —— 技术路线

## 一、系统概述

本系统基于 **Python + Kafka + Spark Structured Streaming + MySQL + Flask + Vue 3** 技术栈构建，实现从交通数据模拟生成、实时流处理分析、到可视化展示的完整链路。

**环境版本：**

| 组件 | 版本 | 说明 |
|------|------|------|
| Python | 3.9+ | 模拟器、Flask 后端 |
| Spark | 3.3.4 | Structured Streaming + Spark SQL（Scala 2.12） |
| Kafka | 3.5.1 | 消息队列 |
| MySQL | 8.0 | 统计结果持久化 |
| Java | 1.8 / 11 | Spark / Kafka 运行环境 |
| Vue 3 | 3.4+ | 前端框架 |
| ECharts | 5.x | 可视化图表 |

## 三、详细技术路线

### 3.1 数据采集层

**Python + confluent-kafka Producer**

- Python 模拟器按每 **5 秒**生成约 **50 条** JSON 格式交通数据
- 通过 `confluent-kafka` 的 Producer API 发送到 Kafka Topic `traffic-raw-data`
- 发送时指定 `acks=1`、`batch_size=16384`、`linger_ms=10` 优化吞吐
- 模拟时段特征：早高峰 7-9 时段 / 晚高峰 17-19 时段 / 夜间 22-6 时段 / 平峰，其余时段流量、速度、拥堵指数各有差异
- 约 15% 概率注入异常数据（空 detector_id / 空 timestamp）

**Kafka Topic 配置（4 个）：**

| Topic | 分区数 | 来源 | 消费方 |
|-------|--------|------|--------|
| `traffic-raw-data` | 6 | traffic_simulator | spark_cleaner |
| `traffic-cleaned` | 6 | spark_cleaner | 所有分析作业 |
| `traffic-alerts` | 3 | anomaly_detect | 预留（告警推送） |
| `traffic-stats` | 3 | 预留 | 预留（统计快照） |

---

### 3.2 实时计算层

**Spark Structured Streaming + Spark SQL**

所有流处理逻辑基于 Spark SQL 引擎运行，通过 `from_json` 解析 Kafka 消息后使用 DataFrame API 执行查询。

#### 3.2.1 数据清洗

消费 `traffic-raw-data` Topic，输出到 `traffic-cleaned` Topic 和 MySQL：

| 步骤 | 使用的算子 / 方法 |
|------|------------------|
| Kafka 消息解析 | `from_json` + Schema 定义（15 个字段） |
| 基础过滤 | `COALESCE(detector_id, '')`、`CASE WHEN` 速度/占用率/拥堵指数越界 |
| 数据去重 | `dropDuplicates(['detector_id', 'timestamp'])` |
| 字段标准化 | `COALESCE`、`TO_DATE`、`date_format` |
| 时间字段 | 添加 `date_key`（DATE）、`hour_key`（VARCHAR）、`etl_time`（TIMESTAMP） |
| MySQL 输出 | pymysql 直写（append 模式） |
| Kafka 输出 | confluent-kafka Producer 回调发送 |

#### 3.2.2 基础统计分析（5 项）

消费 `traffic-cleaned` Topic，输出到 MySQL（30 秒触发，upsert via `REPLACE INTO`）：

| 功能 | 核心算子 |
|------|----------|
| ① 道路车流量统计 | `groupBy` detector_id + 1分钟时间窗口，`SUM`/`AVG`/`MAX`/`MIN` |
| ② 平均车速统计 | `groupBy` detector_id，`AVG`/`STDDEV`/`MAX`/`MIN`/`PERCENTILE_APPROX` |
| ③ 拥堵指数计算 | `CASE WHEN` 复合公式：`(flow/200) * 道路系数 * (0.5 + (1 - speed/80)*0.5)`，道路系数：高速 0.8 / 快速路 0.85 / 主干路 1.0 / 次干路 1.2 / 支路 1.5 |
| ④ 高峰时段分析 | `groupBy` district + date + hour，`CASE WHEN` 分类 peak_type（早高峰 7-9 / 晚高峰 17-19 / 夜间 22-6 / 平峰） |
| ⑤ 区域交通热度 | `approx_count_distinct` + 数学加权公式：`flow/1000*0.3 + avg_congestion*0.5 + (100-speed)/100*0.2` |

#### 3.2.3 进阶分析（4 个已实现 + 4 个待开发）

| 功能 | 核心算子 | 状态 |
|------|----------|------|
| ① 滑动窗口 TopN | `HOP()` 滑动窗口（10min/2min滑动）+ `ROW_NUMBER()` 排名 | 待开发 |
| ② 热点道路 TopN | `ROW_NUMBER() OVER (ORDER BY congestion_score DESC)` 取 Top10 | 已实现 |
| ③ 异常车辆检测 | `LAG()` + `STDDEV() OVER (ROWS BETWEEN 60 PRECEDING)` 计算 Z-Score，过滤 `|z| > 3`，双输出（MySQL + `traffic-alerts` Topic） | 已实现 |
| ④ 交通事故预警 | `ROW_NUMBER()` 连续组编号 + `HAVING COUNT(*) >= 3` | 待开发 |
| ⑤ 拥堵预测 | `LAG(col, 1/3/5)` 多级偏移 + 加权移动平均 | 待开发 |
| ⑥ 区域拥堵排行 | `RANK() OVER (ORDER BY congestion_score DESC)` | 已实现 |
| ⑦ 持续拥堵检测 | `mapGroupsWithState` 状态管理（拥堵持续超阈值时自动延长） | 待开发 |

#### 3.2.4 数据输出

- **到 MySQL**：`foreach_batch_mysql()` 工厂函数封装，通过 `REPLACE INTO` 实现 upsert（先 DELETE 再 INSERT，确保主键唯一）
- **到 Kafka**：`to_json(struct("*"))` + `writeStream.format("kafka")`（仅 anomaly_detect 使用）

#### 3.2.5 状态管理

为避免 Windows 本地文件系统下 HDFSBackedStateStore 的 delta 文件竞争 bug，所有 Spark 作业统一使用 **RocksDBStateStoreProvider**：

```python
.config("spark.sql.streaming.stateStore.stateStoreProviderClass",
        "org.apache.spark.sql.execution.streaming.state.RocksDBStateStoreProvider")
```

**MySQL 8.0 — 9 张表**

| 表名 | 用途 | 关键索引 |

| `region_hourly_stats` | 区域小时统计 + 高峰类型 | 联合唯一: district + date + hour |
| `region_heat_rank` | 区域热度排行 | district + date |
| `hotspot_roads` | TopN 热点道路 | detector_id + period |
| `region_congestion_rank` | 区域拥堵排行 | district + date |
| `traffic_alerts` | 告警事件记录（异常/事故/拥堵） | type + start_time, status |
